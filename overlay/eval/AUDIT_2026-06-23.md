## 1. 🔴 P0 — 指标计算 bug（数字不可信）

### 1.1 `recall_at_k` 实际是 MRR，不是 Recall

**位置**：`rag_eval.py:171-179`

```python
top_k = len(expected_sources)   # 通常 = 1
recall_at_k = 0.0
for i, expected in enumerate(expected_sources[:top_k]):
    expected_pattern = expected.replace('*', '.*')
    for retrieved in retrieved_files[:top_k]:
        if re.search(expected_pattern, retrieved):
            recall_at_k += 1.0 / (i + 1)   # ← 1/(i+1)，i 是 expected 在 expected_sources 里的下标
```

**问题**：
- Recall@K 的定义是 `|relevant ∩ retrieved_top_K| / |relevant|`
- 这里累加的是 `1/(i+1)`，是 MRR 的变体
- 且 `i` 是 expected 在 **expected_sources 列表**里的下标，跟 retrieved 的实际排名无关
- 变量名、累加式、语义三者对不上

**影响**：所有"召回成功率"数字无效。

---

### 1.2 `source_coverage` 给"文件存在"白送 0.5 分

**位置**：`rag_eval.py:66-72`

```python
if glob.glob(full_pattern):
    coverage += 1.0 if matched else 0.5   # ← 文件存在就给 0.5
elif matched:
    coverage += 0.8
```

**问题**：
- expected 文件在项目里存在但 retriever 没命中 → 每个 pattern 白拿 0.5
- `retrieval_success = source_coverage >= 0.5`（line 188）
- 等价于：retriever 啥都不返回，只要 expected 文件存在，就判 success

**影响**：检索"成功率"基本等于"项目里有文件就过"。

- 测的是命名习惯，不是检索质量

### 1.4 `orphaned_pages` 比较的不是同一个东西

**位置**：`ingest_check.py:95-102`


**问题**：

**影响**：孤立页面数不可信。

### 1.5 `parse_frontmatter` 是玩具 YAML 解析器

**位置**：`ingest_check.py:41-46`、`generate_test_cases.py:62-67`

### 9.2 修复清单

#### P0-1 修 `recall_at_k` → 真 Recall@K + 同时报 MRR

**文件**：`overlay/eval/rag_eval.py`

**改法**：
- 把 `recall_at_k` 重命名为 `recall_at_k`（真 Recall），同时新增 `mrr` 字段
- Recall@K = `|relevant ∩ retrieved_top_K| / |relevant|`，其中 K=10（topK 参数）
- relevant = `expected_sources` 用 glob 展开后的实际文件集合
- retrieved_top_K = 检索返回的前 K 个文件路径
- MRR = `1 / rank_of_first_relevant`（第一个相关文档的排名倒数，未命中=0）

**关键点**：
- 用 glob 展开 `expected_sources`，得到**实际文件集合**（不是 pattern 字符串）
- 比对时规范化路径（`os.path.normpath` + 相对项目根）
- K 取 10（跟 search API 的 topK 一致），不是 `len(expected_sources)`

**测试用例**：
- 期望 `[wiki/sources/foo.md]`，检索返回 `[wiki/sources/foo.md, ...]` → recall=1.0, mrr=1.0
- 期望 `[wiki/sources/foo.md]`，检索返回 `[wiki/sources/bar.md, wiki/sources/foo.md]` → recall=1.0, mrr=0.5
- 期望 `[wiki/sources/foo.md]`，检索返回 `[wiki/sources/bar.md]` → recall=0.0, mrr=0.0
- 期望 `[wiki/sources/foo*.md]`（glob），项目里有 foo1.md/foo2.md → relevant={foo1,foo2}，检索返回 foo1 → recall=0.5

#### P0-2 修 `source_coverage` 白送 0.5 逻辑

**文件**：`overlay/eval/rag_eval.py:47-72`

**改法**：
- 删掉"文件存在就给 0.5"的分支
- `source_coverage` 改为纯检索命中比例：`|matched_expected_patterns| / |expected_patterns|`
- matched = 该 pattern 在 retrieved_files 里有匹配
- 文件是否存在不属于评测范围（那是数据准备问题，不是检索问题）

**新语义**：
```python
def check_source_coverage(retrieved, expected, project_dir):
    if not expected:
        return 1.0  # 无期望 = 不考核
    matched = 0
    for pattern in expected:
        regex = glob_to_regex(pattern)
        if any(regex.search(f) for f in retrieved):
            matched += 1
    return matched / len(expected)
```

- `retrieval_success` 阈值从 0.5 改为 1.0（所有 expected 都命中才算成功），或单独报 `coverage` 不再用作 success 判定

**测试用例**：
- expected=`[a.md, b.md]`, retrieved=`[a.md]` → coverage=0.5
- expected=`[a.md]`, retrieved=`[b.md]` → coverage=0.0
- expected=`[]` → coverage=1.0（不考核）

#### P0-4 修 `orphaned_pages` 集合比较

**文件**：`overlay/eval/ingest_check.py:67-104`

**改法**：
- wikilink 目标规范化：去 `|alias`、去 `#anchor`、去 `.md` 后缀

```python
def get_page_identity(md_file, wiki_dir):
    rel = os.path.relpath(md_file, wiki_dir)
    return os.path.splitext(rel)[0]  # 如 "entities/崔玉涛"

def normalize_link_target(link):

# orphan = 文件标识集合 - 链接目标集合（含模糊匹配）
```

#### P0-6 修 `compare_source_to_wiki` 子串误匹配

**文件**：`ingest_check.py:168-205`

**改法**：
- 用**精确文件名匹配**（basename 去后缀完全相等）
- 或用**归一化后的 slug 匹配**（去空格、去标点、小写后比较）
- 不再用 `in` 子串判断

**新逻辑**：
```python
def normalize_slug(name):
    # 去 .md、去标点、去空格、小写
    name = os.path.splitext(name)[0]
    name = re.sub(r'[^\w一-鿿]', '', name).lower()
    return name

# 匹配 = normalize_slug(raw_base) == normalize_slug(wiki_base)
```

**测试用例**：
- `01.md` vs `101.md` → 不匹配（修后）
- `崔玉涛-01.md` vs `崔玉涛01.md` → 匹配（归一化后都是 `崔玉涛01`）
