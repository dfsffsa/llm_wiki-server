# HN 帖子分析：如何用低成本方式搭建网站

分析对象：

- HN 讨论串：https://news.ycombinator.com/item?id=47736555
- 原文：How I run multiple $10K MRR companies on a $20/month tech stack
- 分析日期：2026-04-15

## 1. 这篇文章到底在讲什么

作者 Steve Hanov 的核心观点不是“所有网站都应该极限省钱”，而是：

1. 绝大多数早期产品根本不需要云原生那套复杂基础设施。
2. 对独立开发者或小 SaaS 来说，认知负担往往比机器成本更贵。
3. 与其先做 AWS/K8s/RDS/微服务，不如先做一个能卖、能部署、能稳定运营的单机系统。

他的技术主张可以压缩成一条：

`单机 VPS + Go 单二进制 + SQLite + 可选本地 GPU 跑批处理 AI + OpenRouter 兜底调用闭源模型`

这套组合的优点很明确：

- 成本低，通常可控制在每月 5 到 20 美元级别。
- 部署链路短，出问题容易定位。
- 对 1 人或 2 人团队很友好，几乎没有平台工程负担。
- 对“工具型网站、内容站、轻 SaaS、管理后台、数据查询站”非常适合。

但这篇文章也有明显的表达偏置：

- 它故意拿“最重的企业方案”当反面教材，有一定稻草人倾向。
- 它强调的是“能跑起来且赚钱”，不是“多人协作、极致可靠、复杂业务演进”。
- 有些说法适合作为方向判断，不适合作为绝对规则。

## 2. HN 评论区的真正共识

500+ 条评论里，真正有价值的共识大概是下面几条。

### 2.1 大家普遍认可“早期别过度上云”

这是评论区最强共识。多数人认同：

- 小产品的瓶颈通常不是基础设施，而是获客、留存和迭代速度。
- 很多人确实为几十 RPM 的产品配了过重的架构。
- 一台 VPS 经常足够支撑远超想象的负载。

评论区不少人给了类似反例：

- 用单台 VPS 跑多个 PHP/Laravel/MySQL/Redis 应用。
- 用 EC2 小实例、自管 NAT、Caddy/Dozzle/监控，总成本也只在几十美元。
- 用 Vercel/Neon/Cloudflare 免费层也能先跑起来。

结论：文章的“反复杂度”方向是对的。

### 2.2 最大争议点不是 VPS，而是 SQLite

评论区最集中的技术争论都围绕 SQLite。

支持 SQLite 的观点：

- 单机同进程访问确实非常快。
- 配好 `WAL`、`busy_timeout`、`foreign_keys` 等参数后，读写体验会比很多人想象中好。
- 对单体应用、低到中等写入并发、以业务开发为主的场景，非常够用。
- 备份和复制现在有更成熟的补充方案，例如 Litestream 这类工具。

质疑 SQLite 的观点：

- 多写者、外部 worker、定时任务、webhook 同时写入时，锁竞争依然会暴露出来。
- WAL 不是“并发写入万能药”；它主要改善“读写互斥”，不是把 SQLite 变成 Postgres。
- 当团队规模扩大、运维分工增多、报表和外部分析需求变多时，Postgres 更顺手。

我对这部分的判断：

- “SQLite 适合早期低成本网站”这件事是对的。
- “SQLite 对大多数生产系统都优于 Postgres”这个结论过头了。
- 更准确的说法应该是：
  `单机单体产品的默认起点可以是 SQLite；当写并发、外部集成、团队协作、分析需求提升后，再升级到 Postgres。`

### 2.3 Go 不是唯一解，但“单二进制部署”确实很有价值

评论区有人反驳“Python 完全够用”，这也成立。但支持作者的人抓住了真正重点：

- Go 的关键优势不只是性能，而是部署简单、内存占用小、依赖分发简单。
- 对 1GB 内存 VPS 而言，Go/Rust/Zig 这一类静态编译语言更容易压成本。
- 这不是在说 Python 不能做，而是在说“用更轻的运行时更省心”。

我的判断：

- 如果你是为了“尽快上线”，熟悉 Python/Node/PHP，完全可以继续用。
- 如果你是从零开始、又很在意小内存主机成本，Go 确实是高性价比选择。

### 2.4 作者对 AI 成本的判断只适合“离线批处理”

评论区也有人指出，本地 GPU 不一定真便宜，因为：

- 有前置硬件成本。
- 有电费、散热、稳定性、远程访问和维护成本。
- 对在线、低延迟、波峰波谷明显的用户请求场景，本地 GPU 运维不一定划算。

这点很关键。作者的 AI 方案更适合：

- 批量摘要
- 离线抽取
- 夜间处理任务
- 对隐私敏感的数据处理

不太适合一上来就拿它承接核心在线聊天链路。

## 3. 文中提到的几个产品，分别说明了什么

### 3.1 WebSequenceDiagrams

网站：https://www.websequencediagrams.com/

这是最值得你学习的“低成本建站范式样本”。

我对它的判断：

- 这是典型的窄需求工具站。
- 输入非常简单：一段文本描述。
- 输出也很明确：序列图。
- 价值闭环短，用户能立刻理解和试用。
- 技术形态天然适合单体部署，不需要复杂分布式系统。

为什么这种站适合低成本搭建：

- 计算边界清晰。
- 页面数量少，前台交互不复杂。
- 核心价值集中在一个非常明确的转换能力上。
- 非常适合 SEO、长尾关键词和开发者自传播。

你如果想复制这类路径，重点不是“序列图”，而是：

- 找一个输入输出清晰的窄问题。
- 让用户几秒内看到结果。
- 让系统的大部分成本集中在 CPU/内存可控的同步处理里。

### 3.2 Eh-Trade

网站：https://eh-trade.ca/

从现有公开页面看，它更像一个“股票研究 / 投资内容 + 工具”的垂直产品，博客内容大量围绕：

- momentum investing
- relative strength
- leverage / risk management
- behavioral finance

这个产品透露出的重要信号是：

- 作者做的不是“通用大平台”，而是垂直领域高价值信息产品。
- 这种站点的成本重点不在网页托管，而在数据、模型、内容生产和可信度。
- 网站本身仍然可以非常轻；真正昂贵的是“研究能力”和“持续输出”。

对你有启发的地方：

- 如果你的网站核心是内容、报告、分析、工具输出，而不是复杂多人协同系统，那么单机方案非常合适。
- 真正该投入的不是云账单，而是数据采集、处理流程和转化漏斗。

### 3.3 `smhanov/auth`

仓库：https://github.com/smhanov/auth

这个库体现了作者的一个工程理念：把每个新项目都会重复写的“脏活累活”产品化。

仓库 README 的关键信息：

- 自托管认证系统
- 支持 Email/Password、OAuth、SAML
- 支持 SQLite 和 PostgreSQL
- 内置密码重置、账号创建、会话管理等通用流程

这说明什么：

- 真正让低成本建站更可复制的，不只是“选便宜组件”，而是把重复模块标准化。
- 你如果想长期低成本做站，应该尽早积累自己的：
  - 登录模块
  - 支付模块
  - 管理后台骨架
  - 邮件模板
  - 部署脚本
  - 监控和备份脚本

### 3.4 OpenRouter

网站：https://openrouter.ai/

官网当前强调的信息包括：

- “The Unified Interface For LLMs”
- 一个 OpenAI 兼容接口对接多家模型
- 支持 fallback routing
- 无订阅，买 credits 使用

这类产品的价值在于：

- 把多模型接入统一成一个 API 适配层。
- 减少你在 OpenAI / Anthropic / Google 间反复改 SDK 的工作量。
- 在模型或供应商不可用时更容易做降级。

适合你的使用方式：

- 用它承接低频、但需要高质量模型的用户请求。
- 把“模型选择”从业务代码里剥离出来。
- 但不要把核心业务完全绑定在第三方路由器上，至少保留直接接 OpenAI/Anthropic 的能力。

### 3.5 Ollama

网站：https://ollama.com/

文章里把它放在“本地试验和 prompt 迭代”阶段，这个定位是合理的。

优点：

- 本地跑模型上手快。
- 非常适合开发环境、单人试验、小批量测试。
- 对“先验证流程”非常友好。

限制：

- 高并发和生产吞吐通常不是它的强项。
- 如果你的网站是真正在线承载用户请求，Ollama 更适合研发侧，不一定适合主生产通路。

### 3.6 vLLM

文档：https://docs.vllm.ai/en/latest/

官方文档当前首页强调：

- fast and easy-to-use library for LLM inference and serving
- state-of-the-art serving throughput
- PagedAttention
- continuous batching

这意味着它适合：

- 自己托管开源模型
- 做更高吞吐推理服务
- 把多个并发请求批量压进 GPU

但它并不意味着“更省心”。vLLM 的现实门槛包括：

- 你得有可用 GPU
- 你得处理驱动 / CUDA / 显存 / 模型兼容性
- 你得处理线上稳定性

所以对大多数低成本网站来说，vLLM 不是起步必需品，而是第二阶段优化项。

### 3.7 Transformer Lab

网站：https://lab.cloud/

文章把它定位在更高级的本地训练 / 微调流程。对“先做一个低成本网站”这件事来说，它不是关键路径。

你可以先忽略，除非你的站点强依赖：

- 本地微调
- 专有数据训练
- 多实验版本管理

### 3.8 `laconic` 与 `llmhub`

仓库：

- https://github.com/smhanov/laconic
- https://github.com/smhanov/llmhub

虽然文章对这两个仓库的介绍偏“作者视角”，但从定位上很清楚：

- `laconic` 是作者自用的 agent/research 工具，强调在受限上下文窗口中管理信息。
- `llmhub` 是一层模型 provider 抽象。

它们对你的启发不在于“直接拿来用”，而在于架构思想：

- 把 LLM 接入抽象成一层。
- 把上下文管理、任务编排、模型切换做成独立模块。
- 不要在业务代码里到处散落不同模型供应商逻辑。

## 4. 如果你想做“类似的低成本建站方法”，应该怎么落地

### 4.1 最值得复制的不是具体组件，而是分层原则

建议直接复制下面这套原则：

1. 先单机，后分布式。
2. 先单体，后服务拆分。
3. 先可部署，后可扩展。
4. 先把备份和监控做好，再谈高可用。
5. 只在真实瓶颈出现后升级数据库、队列、对象存储和 AI 基础设施。

### 4.2 一个真正可执行的低成本网站默认方案

如果你今天就要开一个网站，我建议你优先选这套，而不是完全照搬文章：

#### 方案 A：最稳的低成本通用方案

- 1 台 1GB 或 2GB VPS
- `Caddy` 或 `Nginx`
- 应用层：
  - 熟悉 Go 就用 Go
  - 熟悉 Node 就用 Node
  - 熟悉 Python 就用 Python，但尽量保持单体
- 数据库：
  - 默认 `SQLite`
  - 有明显多写者并发时切 `Postgres`
- 静态资源：
  - 初期放本机磁盘即可
  - 图片多了再接 S3/R2
- 备份：
  - 数据库定时快照
  - 每日异地备份
- 监控：
  - uptime 检查
  - 应用日志轮转
  - 基础 CPU / RAM / 磁盘告警

这套方案能覆盖：

- 工具站
- 会员内容站
- 轻量 SaaS
- 后台管理系统
- AI 包装型网站

#### 方案 B：更偏内容站 / SEO 站

- 静态站生成器或 Next.js 静态导出
- 前台静态托管到 Cloudflare Pages / Vercel / Netlify
- 后台管理、抓取和任务系统放一台 VPS
- 数据库可先 SQLite

适合：

- 博客
- 内容聚合站
- 程序化 SEO 页面
- 报告/工具混合站

#### 方案 C：带 AI 能力，但仍然低成本

- 在线用户请求：
  - 先接 OpenRouter / OpenAI API
- 离线批处理：
  - 有 GPU 再考虑 Ollama / vLLM
- 不要一开始就为了“省 token”去搭一整套本地推理集群

这比文章更现实。因为大多数项目的第一目标是验证需求，不是优化 GPU 利用率。

### 4.3 你应该优先补齐的，不是“云架构”，而是这 8 个基础件

低成本建站真正容易出问题的地方不是服务器，而是这些通用能力缺失：

1. 登录与会话
2. 支付与订单
3. 邮件发送
4. 管理后台
5. 日志与错误告警
6. 数据备份与恢复演练
7. 部署脚本
8. SEO / 分析 / 转化埋点

把这 8 个东西做成模板，你以后开新站的速度会明显快于“重新选云产品”。

### 4.4 一个务实的升级路线图

可以按下面顺序升级，不要倒过来。

阶段 1：验证期

- 单 VPS
- 单体应用
- SQLite
- 本机文件存储
- 最小监控

阶段 2：有收入期

- 2GB 到 4GB VPS
- 加对象存储
- 加异地备份
- 加更稳的日志和告警
- 必要时换 Postgres

阶段 3：明确增长期

- 分离后台任务
- 分离数据库
- CDN
- 缓存层
- 灰度部署

阶段 4：团队化运营

- 规范 CI/CD
- IaC
- 更完整权限模型
- 多环境
- 更严谨的数据恢复演练

## 5. 我对你应该怎么做的直接建议

如果你的目标是“类似作者那样低成本搭一个能长期运营的网站”，我建议你不要机械复制全文，而是采用下面这版更稳的做法：

### 推荐的起步组合

- 前端：你熟悉什么就用什么，优先简单
- 后端：Go 或 Node
- 数据库：SQLite 起步
- 反向代理：Caddy 优先，Nginx 也可以
- 部署：单 VPS
- 图片/附件：先本机，后 R2/S3
- AI：先 API，后本地模型

### 什么时候不要用 SQLite

满足下面任一条，就该认真考虑 Postgres：

- 多个 worker 同时写
- webhook / cron / 后台任务频繁写库
- 复杂报表和数据分析需求明显增加
- 团队不止你一个人维护
- 你已经开始担心恢复、迁移和外部接入

### 什么时候不要学作者那套“极简主义”

- 你做的是强协同产品
- 你做的是高写入并发系统
- 你做的是金融/医疗/高审计要求系统
- 你所在团队本来就有成熟云平台和 SRE 支撑

## 6. 最后结论

这篇文章最有价值的地方，不是“Go 胜过 Python”或者“SQLite 胜过 Postgres”，而是它提醒你：

`在产品早期，最该节省的是系统复杂度，不只是服务器账单。`

如果你要实现类似的低成本建站方法，最靠谱的落地路线是：

- 用单机单体起步
- 用最少的外部依赖上线
- 把通用能力做成自己的模板
- 把升级点放在真实瓶颈之后

一句话版建议：

`先搭一个能卖的、能备份的、能恢复的单机网站，再考虑云原生。`
