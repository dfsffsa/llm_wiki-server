# 作者前端实现方案分析

分析日期：2026-04-15

研究目标：

- 看作者几个网站的前端实现方案到底是什么
- 判断他是否真的在前端层做到了“极致轻量”
- 提炼对你有价值的前端性能策略

研究对象：

- `websequencediagrams.com`
- `eh-trade.ca`
- `stevehanov.ca/blog`

---

## 1. 先说结论

最重要的结论是：

`这位作者真正优化的不是“所有页面的前端 bundle 都超小”，而是“把重前端严格限制在必须重的页面，把真正费资源的工作移回后端”。`

也就是说，他的高性能不是靠一条原则，而是三条组合：

1. 能做服务端渲染 / 纯 HTML 的页面，就尽量不用前端框架。
2. 必须做重交互的页面，只把前端变成一个很窄的工作台，不把业务计算放浏览器里。
3. 真正吃 CPU / 内存的事放后端，前端只负责输入、展示和少量状态管理。

所以如果你问：

“他是不是所有网站都前端很轻？”

答案是：

- `博客` 很轻
- `WebSequenceDiagrams` 是重交互单页应用，但只在一个非常窄的场景里重
- `eh-trade` 的前端其实不算特别轻，甚至相对偏重

这说明他追求的是：

`系统整体简单和高效`

而不是：

`前端永远最小 bundle`

---

## 2. 站点逐个分析

## 2.1 Steve Hanov 的博客：明显是“服务端优先”

博客页面当前公开结构很直接：

- 直接输出完整 HTML
- 独立 CSS：`/static/blog-custom.css`
- 只引入了老版 jQuery
- 评论区是页面内增强，不是前端 SPA 接管整站

从抓到的资源体积看：

- HTML 大约 `37,568` 字节
- CSS 大约 `18,086` 字节
- jQuery 约 `94,840` 字节

这个博客的前端方案很像传统网站：

- 首屏直接可见
- HTML 天然适合 SEO
- 没有客户端 hydration
- 页面切换和内容阅读几乎没有复杂状态

这恰恰是“高性能独立开发者前端”的一个核心手法：

`不是所有页面都值得上 SPA。`

对于博客、文档、营销页、文章页，他明显选择了：

- 简单 HTML
- 少量 CSS
- 极少量 JS 增强

这才是真正的低内存、低复杂度。

## 2.2 WebSequenceDiagrams：这是重前端，但重得很克制

从首页 HTML 可以确认：

- 页面是单页应用
- 使用本地 `Ace Editor`
- 主 bundle：`/assets/index-DEN24LnG.js`
- CSS：`/assets/index-CF030y3w.css`

抓到的资源体积大致是：

- 主 JS：`903,465` 字节
- Ace：`446,668` 字节
- CSS：`55,990` 字节

这说明它前端并不轻到离谱。因为编辑器类产品本来就需要：

- 代码编辑器
- 实时交互
- 状态管理
- 预览控制

但关键在于，它的“重”是有边界的。

### 证据 1：它的重前端只服务于一个核心工作台

WSD 的产品形态非常集中：

- 左边输入文本
- 右边生成图
- 周边是保存、文件、分享、订阅

它不是一个到处复杂跳转的大型应用壳，而是一个非常窄的编辑器工作台。

这类页面即使前端不轻，也能保持体验快，因为：

- 用户心智单一
- 页面结构单一
- 状态空间相对有限

### 证据 2：真正的渲染工作不在浏览器里

从前端 bundle 里能看到一个非常关键的信号：

- 它会请求后端 `index.php`
- 返回 diagram 结果和图片 URL

这说明：

`图生成的核心工作并不是在浏览器里完成，而是在后端完成。`

这非常关键。

如果作者把 sequence diagram 的布局和渲染全塞进浏览器：

- bundle 会更大
- 浏览器 CPU 会更吃紧
- 前端代码会更复杂
- 稳定性更难控制

他显然没有这么做。

他的方式更像：

- 浏览器负责编辑器体验
- 服务器负责解析和生成最终图像

这是一种典型的独立开发者高性能策略：

`让前端重交互，但不让前端承担重计算。`

### 证据 3：他选的是 Ace，不是更重的在线 IDE 栈

WSD 用的是 Ace。

这也透露出作者的偏好：

- 功能够用
- 嵌入简单
- 比一整套更现代的 IDE 方案更可控

Ace 不算轻，但它是“合适的重”，不是“为了技术时髦而重”。

## 2.3 eh-trade：前端其实偏重，不是极致简化路线

从首页和 bundle 可以确认：

- 是 SPA
- 明显使用 Vue
- bundle 里能看到 `pinia`
- bundle 里能看到 `vue-router`
- 页面交互、动画、营销层、数据展示都较多

抓到的资源体积大致是：

- 主 JS：`1,264,681` 字节
- CSS：`389,752` 字节

这已经不能算“极致轻前端”了。

所以如果单看 eh-trade，你会得出一个很重要的判断：

`作者并不是教条式地反对现代前端。`

他更像是：

- 需要营销和产品表达时，可以接受一个偏重的 Vue SPA
- 但真正吃资源的研究、选股、AI 分析不在浏览器里做

这点尤其关键。因为 eh-trade 的核心价值不是页面切换，而是：

- 数据筛选
- 研究结果
- AI 分析
- 风险判断

这些东西如果都堆到前端，才会真的变慢。

而作者实际上把重逻辑放回了后端和离线处理链路。

---

## 3. 他前端性能真正好的原因

## 3.1 他不是在所有地方追“现代前端最佳实践”

这位作者的一个明显特征是：

- 页面类型不同，前端策略完全不同

具体是：

- 博客：传统 HTML + CSS + 少量 JS
- 编辑器：窄场景 SPA + 编辑器组件
- 产品站：SPA，但核心计算在后端

这比“全站同一种技术栈”更省资源。

很多团队的问题恰恰是：

- 博客也做成 SPA
- 登录页也做成 hydration 大应用
- 营销页也塞进统一大前端壳

作者明显没有这么做。

## 3.2 他把真正重的逻辑放回后端

这是最关键的性能来源。

WSD 的图生成在后端。
eh-trade 的研究和分析在后端 / 离线流程。

这样浏览器只承担：

- 输入
- 展示
- 小范围状态
- 网络请求

于是前端即使不是最小 bundle，用户主观体验依然可能很好。

因为最容易让页面“卡”的不是单纯 bundle 大，而是：

- 浏览器里做复杂计算
- 大量客户端数据处理
- 大量实时重排和重绘
- 过度的前端状态同步

作者基本避免了这些。

## 3.3 他让“重页面”变成少数特例，而不是全站默认

WSD 的重前端是因为编辑器确实需要。
博客没有被连带拖成 SPA。

这说明他的默认思路是：

- 先问页面到底需不需要高度交互
- 只有真的需要，才上重前端

这非常值得学。

## 3.4 他的前端并不追求复杂工程体系

从可观察信息看，他没有明显暴露出这些“现代大厂默认包袱”：

- 前端微服务
- 超复杂 SSR/ISR/RSC 架构
- 前后端分治的庞大 BFF 体系
- 多层 hydration 方案

这会直接减少：

- 维护负担
- 构建复杂度
- 内存占用
- 线上问题定位成本

---

## 4. “极致性能”应该如何正确理解

如果你把“极致性能”理解为“最小 JS bundle”，你会误判他。

更准确的理解应该是：

`他做的是系统级性能优化，而不是前端教条式极简。`

系统级性能包括：

1. 页面类型分层
2. 计算位置正确分配
3. 浏览器只负责 UI，不负责重业务计算
4. 简单部署，减少中间层
5. 让前端复杂度和产品价值成比例

所以：

- 博客轻，是因为博客应该轻
- 编辑器不轻，但它“只在必要的地方重”
- 数据产品前端可重一点，但核心计算不在前端

---

## 5. 对你最有价值的可复用策略

如果你也想走作者这条低成本路线，前端最值得复制的不是某个框架，而是这套决策规则。

## 5.1 页面先分三类

### A. 内容页 / 营销页 / 博客页

建议：

- 直接 HTML 输出
- 少量 CSS
- 极少 JS

不要默认上 SPA。

### B. 工作台页 / 编辑器页

建议：

- 允许用 Vue/React SPA
- 但必须限制在核心工作区
- 重计算尽量后端做

### C. 数据看板 / 会员页

建议：

- 可以用 SPA
- 但只负责状态和展示
- 不要把筛选、分析、AI 推理主逻辑压浏览器

## 5.2 把“算力”从前端拿掉

如果页面真正耗资源的地方是：

- AI
- 图生成
- 复杂布局
- 搜索和评分
- 大数据筛选

优先后端化。

前端只拿结果。

这是最有效的性能优化。

## 5.3 重前端只出现在真正赚钱的页面

WSD 的编辑器值得重，因为那就是核心产品。
博客不值得重，因为读文章不需要那套能力。

这个判断非常重要。

## 5.4 不要为了现代而现代

作者并没有表现出“必须 Next.js / 必须 SSR / 必须复杂 hydration”的执念。

这对独立开发者很重要。

更好的原则是：

- 能不用复杂前端就不用
- 必须用时，只在局部用
- 前端只承担自己最擅长的那一层

---

## 6. 最终结论

作者前端性能好的真正原因，不是“所有站都超轻”，而是：

- 旧式内容页就老老实实做成 HTML 页
- 真正需要交互的地方做窄场景 SPA
- 让浏览器负责 UI，不负责重计算
- 让复杂度只出现在最有商业价值的页面

一句话总结：

`他优化的是“前端在整个系统里该干什么”，而不是盲目追求所有前端都最小。`

如果你要复制他的路线，最值得抄的不是具体框架，而是这条规则：

`内容页尽量无框架，工作台页允许 SPA，但重计算一律后端化。`
