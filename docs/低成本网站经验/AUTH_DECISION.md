# Auth 方案决策记录

日期：2026-04-16

## 决策结论

认证方案正式采用：

- `smhanov/auth`
- GitHub: https://github.com/smhanov/auth

本地自研草稿方案：

- `boringauth`

已删除，不再作为正式实现方向保留。

## 为什么做这个决策

经过前面的分析，`smhanov/auth` 已经不是一个“思路展示”级别的仓库，而是一套完整、公开、可复用的认证内核，已经具备：

- Email/Password 登录注册
- session cookie
- forgot/reset password
- SQLite / PostgreSQL 支持
- OAuth
- SAML
- rate limiting
- schema
- 文档和测试

而本地的 `boringauth` 只是基于讨论快速起的一版骨架，它的价值主要在于：

- 帮助澄清设计边界
- 帮助整理我们偏好的 API 形态
- 作为后续适配层设计参考

但它不适合继续作为独立正式方案推进，原因如下：

1. `smhanov/auth` 已经覆盖了绝大多数核心能力。
2. 认证系统本身安全和稳定要求高，不值得无谓重复造轮子。
3. 同时维护两套认证内核会制造长期分叉成本。
4. 本地骨架没有经过完整编译、测试和生产验证。

因此，最合理的处理方式是：

- **上游 `smhanov/auth` 作为认证内核**
- **我们自己的工作集中在“接入、适配、裁剪、封装”**

## 本地 `boringauth` 的处理结果

已执行：

- 删除原工作目录中的 `boringauth/` 子目录

不再保留它的原因：

- 避免误导后续开发，把半成品误当成正式基础设施
- 避免后续讨论里继续把它和上游并列
- 收敛决策，减少分叉

## 现有文档如何定位

保留这些文档，作为背景分析和设计参考：

- `AUTH_ANALYSIS.md`
- `AUTH_SYSTEM_DESIGN_FOR_LLM.md`
- `AUTH_IMPLEMENTATION_PROMPT.md`

这些文档的用途现在变为：

- 帮助理解 `smhanov/auth` 的设计边界
- 帮助判断哪些地方需要二次封装
- 帮助定义我们理想中的外部 API 形态

它们不再意味着“我们要独立维护另一套 auth 内核”。

## 后续推荐路线

接下来不应该再做“重写一套 auth”，而应走下面这条路线：

### 路线 A：直接采用上游

适用于：

- 新项目快速接入
- 尽快上线
- 不急着统一自定义 API 风格

做法：

- 直接引入 `github.com/smhanov/auth`
- 使用其默认 `/user/*` 路由
- 在业务层扩展 `GetInfo`

### 路线 B：上游内核 + 我们自己的薄适配层

这是目前最推荐的方式。

适用于：

- 想保留更统一的 `/auth/*` JSON API 形态
- 想让前端接口风格更稳定
- 想把业务侧 auth 接入方式标准化

做法：

- `smhanov/auth` 继续做底层认证内核
- 我们自己加一层很薄的 adapter
- adapter 负责：
  - 路由整形
  - JSON 包装
  - 用户信息整合
  - 对业务项目暴露一致接口

### 路线 C：fork 上游

适用于：

- 明确要长期控制 auth 代码
- 需要做有边界的深度改造
- 不希望未来完全受上游接口约束

建议：

- 只做有限改造
- 不轻易重写核心 session / reset / OAuth 流程
- 保持和上游尽量接近

## 当前明确不做的事

1. 不再继续扩写 `boringauth`
2. 不再维护两套正式认证方案
3. 不把 auth 作为“重新发明一遍”的方向推进

## 一句话总结

最终认证策略确定为：

`采用 smhanov/auth 作为正式认证内核，本地自研骨架停止推进，后续只做接入、适配和必要封装。`
