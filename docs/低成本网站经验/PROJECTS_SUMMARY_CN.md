# 1. eh-trade 项目

## 项目链接

- 官网：https://eh-trade.ca/
- 博客：https://eh-trade.ca/blog/

## 这是一个什么项目

从公开页面和首页文案看，`eh-trade` 是一个面向股票/期权/动量交易场景的研究与工具型产品，核心卖点大致是：

- 股票筛选
- 市场研究
- 风险管理
- 交易辅助判断
- 可能还包含订阅制的 premium 数据和分析结果

从首页和前端 bundle 能看到一些能力入口：

- screener
- stock detail
- watch positions
- pricing / upgrade
- portfolio monitoring
- backtester
- power calculator

所以它不是单纯博客，更像：

`投资研究 + 数据工具 + 会员订阅`

## 计算量在哪里

这个项目的重计算，主要不在前端，而在后端和离线链路。

### 公开证据可以确认的

作者在原文中明确说过，做 `eh-trade` 时，他要处理的是：

- 对大量公司做深度定性研究
- 总结大规模季度报告
- 批量处理很多公司材料

这是典型的重计算任务，因为它包括：

1. 大量文本读取与切分
2. 批量摘要与信息提取
3. 多轮 prompt 流程
4. 成本和吞吐控制

### 为什么“单纯数据计算”也会用上大模型

这个问题很关键。

如果只是传统数值计算，比如：

- 涨跌幅
- 波动率
- 回测收益率
- 指标排序

这些当然不需要大模型。

但 `eh-trade` 不是只做数值计算。它至少还涉及：

- 财报文本阅读
- 研究摘要
- qualitative analysis（定性分析）
- 从长文中抽取“企业发生了什么、风险在哪里、亮点是什么”

这些东西传统数值算法很难直接搞定，所以大模型的作用不是代替数据计算，而是补足：

- 文本理解
- 文本压缩
- 信息抽取
- 解释型输出

更准确地说，`eh-trade` 里大模型更像是：

`研究分析层`

而传统代码负责：

`数据抓取、指标计算、筛选、排序、回测、结果组织`

## 这些重计算是怎么处理的

### 公开证据可以确认的

作者明确写了：

- 本地先用 `Ollama` 试模型和 prompt
- 正式批处理阶段用 `vLLM`
- 运行在自己本地 GPU（他文中举的是 24GB 显存的 RTX 3090）
- 用户面向的低延迟交互不一定走本地模型，而是可以走 `OpenRouter`

这说明他的处理策略是：

1. **离线批处理**：本地 GPU + `vLLM`
2. **实验调试**：`Ollama`
3. **用户侧即时高质量回答**：云模型 / OpenRouter

## 离线批处理完成后，前端只是拿结果吗

我的判断是：**大概率是的，而且这正是合理做法。**

### 公开证据可以确认的

从前端 bundle 里看，首页和应用层更多是在：

- 展示筛选结果
- 展示价格、收益、pick
- 输入 ticker
- 发请求拉结果

看不到“浏览器端做大规模 AI 分析”的迹象。

### 更稳妥的结论

`前端主要是拿后端已经整理好的结果，而不是自己做重分析。`

这很符合作者整体风格：

- 浏览器只负责输入和展示
- 重计算放后端
- 批处理放离线链路

## 有实时计算部分吗？还是没有实时，千人一面？

这里要分开看。

### 可以较有把握地判断的

#### 1. 公共首页部分很可能是“千人一面”或接近千人一面

比如：

- top momentum picks
- 营销展示的案例和结果
- landing page 上展示的示例 ticker

这些内容非常像：

- 定时更新
- 统一输出
- 所有人看的是同一批公开 pick

#### 2. 登录后部分很可能存在“用户相关”的结果

前端里能看到：

- 用户状态
- 订阅状态
- dashboard
- watch positions

这说明它并不只是完全静态统一页面。

所以更可能的结构是：

- **公开区域**：大部分统一结果，偏“千人一面”
- **用户区域**：基于用户 watchlist / portfolio / subscription 的个性化展示

#### 3. “实时”更可能是轻实时，不像高频交易那种真实时

目前公开信息看不到它在做：

- 毫秒级实时推理
- 高频 tick 级别前端计算
- 浏览器侧实时 AI 分析

因此我更倾向于判断：

- 它可能有定时刷新 / 近期数据更新
- 可能有用户请求触发的即时计算
- 但主分析结果很可能不是毫秒级实时生成

一句话总结：

`eh-trade 更像“离线分析 + 在线查询/展示 + 局部用户定制”，而不是“所有分析都实时生成”。`

# 2. WebSequenceDiagrams 项目

## 项目链接

- 官网：https://www.websequencediagrams.com/
- 博客：https://www.websequencediagrams.com/blog
- 手册：https://www.websequencediagrams.com/manual.html

## 这是一个什么项目

这是一个非常经典的窄场景开发者工具站：

- 输入一段文本描述
- 输出 sequence diagram

它的价值非常直接：

- 输入简单
- 输出清晰
- 用户几秒内就能看到结果

这类产品非常适合低成本长期运营。

## 项目的后端在哪里部署

### 公开证据可以确认的

从当前前端 bundle 可以明确看到：

- 前端请求同域名下的 `index.php`
- 还请求同域名下的 `/files/*`、`/users/*` 等接口

这说明：

- 前端和后端至少从用户视角上是**同域部署**
- 前端不是在一个站点、后端在另一个 API 域名上

### 更稳妥的推断

大概率是：

- 前端静态资源和后端服务挂在同一个主域名
- 可能由同一台服务器直接提供
- 或者由同一个反向代理统一转发到不同服务

但目前**没有公开证据能 100% 证明**它今天一定物理上就是单机单实例。

## “同域访问”不等于“同一台机器”

这个点很容易混淆，单独说清楚。

对浏览器来说，WSD 的前后端关系看起来是：

- 前端页面来自 `https://www.websequencediagrams.com`
- 前端请求的后端接口也来自同一个域名

因此：

`浏览器只知道同一个域名，并不知道后端到底在哪台机器上。`

也就是说，“前端怎么找到后端”的答案并不是：

- 记住某台机器 IP
- 或直接访问某个内网地址

而是：

- 前端始终请求同一个公网域名
- 域名背后的入口层负责把请求转发到真正的后端

这个入口层可能是：

- nginx
- caddy
- 负载均衡器
- tunnel 入口
- 内网代理

## 可能的三种部署方式

下面这三种对 WSD 这类产品都成立。

### 方案 A：前后端同机

这是最简单、也最符合作者整体风格的方案。

结构：

- 一台有公网 IP 的 VPS
- 静态前端文件
- 后端进程
- 同一台机器上的 nginx / caddy
- 同一个域名

请求链路：

1. 浏览器访问 `https://www.websequencediagrams.com`
2. 反向代理返回前端静态文件
3. 前端发请求到同域后端接口
4. 反向代理把请求转发到本机 `127.0.0.1:xxxx`
5. 后端生成结果并返回

这是最适合独立开发者的第一阶段方案。

### 方案 B：前端在公网机器，后端在另一台无公网 IP 机器

这也完全可行。

结构：

- 一台有公网 IP 的入口机器
- 一台没有公网 IP 的后端渲染机器
- 两台机器通过内网/VPN互通
- 域名仍然只指向入口机器

常见连法：

- Tailscale
- WireGuard
- ZeroTier
- 自建内网专线

请求链路：

1. 浏览器请求 `https://www.websequencediagrams.com/render`
2. 公网入口机器收到请求
3. 入口机器把请求转发到内网后端机器
4. 后端机器生成图并返回
5. 入口机器再把结果回给浏览器

这时前端仍然**不知道**后端在哪台机器上，只知道域名。

### 方案 C：无公网 IP 的后端机器通过 tunnel 暴露

如果后端机器：

- 没有公网 IP
- 也没有端口映射
- 但可以主动访问外网

那么也可以通过 tunnel 方式工作。

常见方式：

- Cloudflare Tunnel
- frp
- Tailscale Funnel
- 各种反向隧道

结构：

- 前端和公网入口挂在一个公共域名上
- 后端机器主动连到公网入口
- 公网入口收到请求后，再通过 tunnel 转发给后端

请求链路：

1. 浏览器请求域名
2. 域名到公网入口
3. 公网入口通过已建立的 tunnel 找到后端机器
4. 后端返回结果

这里仍然不是“前端自己找到后端”，而是“公网入口帮前端找到后端”。

## 这 1GB 内存的 VPS 能部署一个后端吗

可以。

而且对这类项目来说，这并不夸张。

因为 WSD 的后端任务虽然是“生成图”，但它不是：

- 大模型推理
- 大数据分析
- 大规模多租户协作平台

它更像是：

- 文本解析
- 图布局
- 图像导出

这类后端只要实现得不离谱，1GB VPS 完全是可行的。

### 但要注意一个现实点

作者文章里强调的是他现在做项目偏好 Go 单体 + SQLite，但 WSD 是一个存在多年的老产品。

从今天公开前端请求的接口名看：

- 有 `index.php`
- 有一批偏历史风格的接口

所以更合理的判断是：

- WSD 的**当前线上代码不一定是“最新那套 Go 默认模板”**
- 它更可能是一个历史上持续演化过的产品

这不影响你的落地结论：

- 这种产品完全可以放在小 VPS 上
- 前后端同域是可行的
- 低内存服务器也能承担

## 如果后端单独部署，需要一个公网 IP 吗

### 从工程角度说

**不一定。**

如果你的前端和反向代理在公网，后端完全可以：

- 只在内网监听
- 不暴露公网 IP
- 由同机 nginx / caddy 转发
- 或由内网网关转发

### 如果是极简部署

很多独立开发者会直接：

- 一台 VPS
- 一个公网 IP
- nginx / caddy
- 前端静态资源 + 后端进程都挂同一台机器

这种情况下：

- 服务器本身有公网 IP
- 但你的后端进程不一定需要“额外一个独立公网 IP”

一句话总结这个问题：

`后端机器本身可以没有公网 IP，只要有一个公网入口层能够把同域请求转发到它。`

## 请把前后端交互搞清楚，方便我们落地

这是最值得总结的部分。

### 当前公开证据能确认的交互方式

WSD 的前端是一个编辑器工作台，但核心图生成不是前端完成。

从 bundle 可见：

- 前端负责编辑器
- 前端通过请求后端生成图
- 后端返回 diagram 结果和图片资源

这意味着它的交互模式大致是：

1. 前端收集用户输入的 diagram DSL
2. 前端发请求到同域后端
3. 后端做解析与图生成
4. 后端返回图片地址 / 结果信息
5. 前端把结果渲染出来

### 对你最有价值的落地结论

如果你后续做类似项目，我建议直接照这个方向走：

#### 前端负责

- 文本输入
- 编辑器体验
- 参数切换
- 预览刷新
- 导出按钮
- 保存与分享入口

#### 后端负责

- DSL 解析
- 布局计算
- SVG / PNG 生成
- 存储
- 分享链接
- 导出

#### 不建议前端负责

- 图布局核心算法
- 重渲染逻辑
- 大量导出处理

一句话说清楚：

`前端是工作台，后端是渲染引擎。`

这就是它值得借鉴的地方。

## 如果你后续自己做类似产品，我建议的部署顺序

### 第一阶段：同机部署

建议直接做：

- 一台有公网 IP 的便宜 VPS
- 前端静态资源 + 后端进程都放这台机器
- nginx / caddy 做同域转发

原因：

- 最简单
- 最容易排障
- 最容易先跑通

### 第二阶段：后端吃资源时再拆

如果后端渲染明显更重，再演化为：

- 公网入口仍然在 VPS
- 后端渲染服务搬到另一台机器
- 两台机器用 Tailscale / WireGuard 互通
- 入口机器把 `/render` 这类路径转发到后端渲染机器

这样就能实现：

- 域名不变
- 用户无感知
- 后端机器可以没有公网 IP

这通常比一开始就设计复杂的分布式前后端更适合独立开发者。

# 3. laconic 项目

## 项目链接

- GitHub：https://github.com/smhanov/laconic

## 这是什么项目

`laconic` 是一个用 Go 写的 research agent / 小上下文研究代理库。

它的目标非常明确：

- 让小上下文模型（4k / 8k）也能做研究型任务
- 通过压缩状态，避免 agent trace 无限膨胀
- 降低多轮搜索和综合的成本

README 里一句话总结得很准：

`A tiny Go library for building research agents that stay fast and cheap on small-context models.`

## 它主要解决哪些问题

你问的几个问题，它几乎都在正面处理：

### 1. agent trace 越来越长

传统 ReAct agent 的问题是：

- 一轮一轮把历史 trace 全塞回 prompt
- 越跑越长
- 最后溢出上下文

`laconic` 的思路不是保留完整 trace，而是**每轮压缩状态**。

### 2. prompt 膨胀

它采用的核心方法是：

- `scratchpad`
- `graph-reader`

两者都不是无脑叠上下文，而是把“已知信息”压缩成更小、更结构化的表示。

### 3. 小上下文模型很快失控

它的 README 明确强调：

- 设计目标就是 small-context models
- 特别适合 4k / 8k 模型

本质做法是：

- 不把原始搜索结果长期保留
- 只保留压缩后的知识
- 或只保留 atomic facts

### 4. 多轮搜索 / 提取 / 综合成本很高

它从几个层面降低成本：

- 限制迭代次数
- 允许规划模型和综合模型分开
- 支持便宜模型做更多步骤
- 结果里直接记录 `Cost`
- 支持 knowledge carry-over，复用上轮收集到的信息

## 它是如何具体解决这些问题的

README 里公开讲得很清楚，核心就是两种策略。

## 3.1 Scratchpad 策略

这是一个：

`Planner -> Search -> Synthesizer -> Finalizer`

的循环。

关键点在于：

- 每轮搜索后，不保留完整原始结果
- 由 Synthesizer 把旧 knowledge + 新结果压缩成一段更短的知识摘要
- 下一轮只带这个压缩后的知识走

所以它不是不断 append，而是不断 rewrite knowledge。

这直接抑制了：

- trace 无限增长
- prompt 越滚越大

## 3.2 Graph Reader 策略

这个更适合复杂问题。

它不是只保留一团 summary，而是：

- 先做 rational plan
- 拆 key elements
- 搜索
- 提取 atomic facts
- 去重
- 形成 notebook
- 判断是否已经足够回答
- 再决定要不要继续扩展 query graph

这相比“无脑喂全文”有两个优势：

1. 重要信息不会因为 summary 过粗而丢失
2. 仍然避免把整堆网页原文塞进上下文

## 能找到公开实现吗

能。

而且这点对你最有价值。

### 公开能看到的实现层面

从 README 已经能确认：

- 两种 strategy 都是公开的
- 有完整 Go 库
- 有 example
- 有 CLI demo
- 有可插拔 search provider
- 有可插拔 fetch provider
- 有 planner/synthesizer/finalizer 这类角色划分

这不是一篇空想博客，而是已经被做成了库。

### 对你后续需求的借鉴意义

如果你后续也有类似需求，这个项目最值得你学的不是某个 prompt，而是这三条：

1. **不要保留完整 trace**
2. **把知识压成 notebook / scratchpad / atomic facts**
3. **让“继续搜索”成为一个带边界的决策，而不是无限循环**

一句话总结：

`laconic 的核心价值，不是“更会搜”，而是“更会压缩研究过程”。`

# 4. llmhub 项目

## 项目链接

- GitHub：https://github.com/smhanov/llmhub

## 这是什么项目

`llmhub` 是一个统一的、provider-agnostic 的 Go LLM client。

它的作用是：

- 把 OpenAI
- Anthropic
- Gemini
- Ollama
- 或其他 provider

统一到同一个 API 抽象之下。

它支持的公开能力包括：

- 多 provider 切换
- streaming
- multimodal
- provider registry
- functional options
- 成本核算

一句话说：

`它是 LLM 接入层，不是最终产品。`

## 它怎么盈利

### 公开证据能确认的

目前没有看到它有：

- 独立官网定价
- 托管版服务
- SaaS 收费页面

所以从公开信息看，**没有明显直接商业化迹象。**

### 更合理的推断

它更像是：

1. 作者自用的基础设施库
2. 为自己的产品减少 provider 耦合
3. 顺手开源

如果谈“盈利”，它更可能是**间接盈利**：

- 降低作者自己产品的开发成本
- 给作者的其他项目提供统一 LLM 接入层
- 未来可能作为技术品牌、口碑、工具链积累的一部分

所以你现在更值得借鉴的是：

- 统一 provider 接口
- 降低业务代码耦合
- 让模型切换和 endpoint 切换足够便宜

而不是去学它“怎么卖”。

# 5. auth 项目

## 项目链接

- GitHub：https://github.com/smhanov/auth
- Go 文档：https://pkg.go.dev/github.com/smhanov/auth

## 这是一个公开项目吗

是，明确公开。

而且不是半成品，它公开了：

- README
- 文档
- 路由
- schema
- SQLite / PostgreSQL 支持
- OAuth / SAML 支持

这说明它是作者认真抽出来的可复用认证内核。

## 这项目怎么盈利

### 公开证据能确认的

和 `llmhub` 类似，目前也没有看到：

- 独立收费站
- 托管版 auth 平台
- SaaS pricing

所以目前看不到直接收费模式。

### 更合理的推断

它的价值更可能是**间接盈利**：

1. 降低作者自己的新项目启动成本
2. 把重复的认证逻辑沉淀成可复用资产
3. 通过开源建立技术影响力
4. 未来可能间接带动其他产品和咨询机会

从作者文章语境看，这个项目最核心的商业价值不是“单独卖 auth”，而是：

`让他以后做新产品时，登录这件事几乎变成 30 行接入。`

这比单独卖一个 auth SaaS 对独立开发者更现实。

## 对你怎么为己所用

你最值得学它的地方有三点：

### 1. 把认证做成公共底座

而不是每个项目重写：

- 注册
- 登录
- session
- 忘记密码

### 2. 服务端 session 优先

对单体网站来说：

- 简单
- 安全边界清楚
- 撤销会话方便

### 3. auth 只管身份，不吞掉业务

这点很重要。

业务项目自己再扩一层：

- profile
- subscription
- entitlements
- workspace/project ownership

这也是我前面建议你做自己认证内核时最应保留的边界。

# 最后总结

如果把这几个项目合起来看，最值得你吸收的经验其实非常统一：

## 1. 重计算一律后端化

- `eh-trade`：研究分析和批处理在后端 / 本地 GPU
- `WebSequenceDiagrams`：图生成在后端
- `laconic`：重成本通过压缩状态来控制，而不是无限堆上下文

## 2. 通用能力都做成基础设施

- `auth`：认证底座
- `llmhub`：模型接入底座
- `laconic`：研究型 agent 底座

## 3. 前端只是工作台，不是算力中心

这是这位作者非常稳定的一条路线。

一句话总结：

`他真正厉害的地方，不是某一个项目，而是把“重计算、通用能力、前端边界”这三件事分得非常清楚。`
