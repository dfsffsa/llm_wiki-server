# 新项目启动指引

> 最后更新：2026-06-06  
> 适用仓库：`llm_wiki-server`（HTTP 服务 + CLI + 桌面 upstream）  
> 示例项目：`~/overseas-github/llm_wiki_projects/CivilCareer`（已跑通）、`ParentingBooks`（育儿书，本文示例）  
> **已有项目的日常维护**（加材料、Ingest、切换项目）见 [日常运维.md](./日常运维.md)  
> 全局文档入口：[文档指引.md](../文档指引.md)

当你有一批原始材料（Markdown 章节、PDF、txt 转写等），希望转成 LLM Wiki 能检索、能问答的知识库时，按本文从 0 到 1 操作。

---

## 0. 先搞清楚两件事

### 0.1 LLM Wiki 在做什么

不是「把文件丢进去就能 Chat」的普通 RAG，而是 **Karpathy LLM Wiki 模式**：

```text
原始材料 (raw/sources/)  →  Ingest（LLM 编译）  →  结构化 wiki/  →  搜索 / 提问
```

| 阶段 | 含义 |
|------|------|
| **Ingest** | LLM 读原始章节，生成 `wiki/sources/`、`concepts/`、`entities/` 等结构化页面 |
| **Query** | 在 `wiki/` 上检索，再让 LLM 带引用回答 |
| **Lint** | 检查断链、孤儿页、结构一致性（可选） |

### 0.2 HTTP 页面 vs 桌面版：能力不同

| 能力 | HTTP（`llm-wiki-server`） | 桌面版（`upstream` + `tauri dev`） |
|------|---------------------------|-------------------------------------|
| 浏览 wiki / 搜索 / 图谱 | ✅ | ✅ |
| **Chat 对话提问（RAG）** | ✅（服务端 LLM 代理，需 `LLM_WIKI_CONFIG`） | ✅ |
| Ingest 入库 | CLI | GUI 队列 + CLI |
| 写入 wiki | ❌ | ✅ |
| Web Search / AnyTXT（Chat 附加源） | ❌（首期） | ✅ |

**结论：**

- **HTTP 部署**现已支持：浏览、搜索、**Chat 提问**（LLM 经 server 代理，API Key 不暴露给浏览器）。
- **桌面版**仍适合：入库 GUI、写入 wiki、完整 Settings 与多源 Chat。

配置 Chat：启动 server 时设置 `LLM_WIKI_CONFIG`（须含 `llmConfig`），并导出 `LLM_API_KEY` 等环境变量。

更多背景见 [开发与测试.md §Q5](./开发与测试.md#q5像桌面-llm_wiki-那样在页面上测试大模型-chat-怎么做)。

## 1. 判断你的材料是否已是「项目格式」

### 1.1 标准项目目录

创建项目时会生成（见 upstream `create_project`）：

```text
<project-root>/
├── purpose.md              # 项目目标（给 Ingest LLM 看）
├── schema.md               # 页面类型与提取规则（给 Ingest LLM 看）
├── raw/
│   ├── sources/            # 原始材料（只读，immutable）
│   └── assets/             # 图片等资源
├── wiki/                   # LLM 编译后的知识页（检索与问答都在这里）
│   ├── index.md
│   ├── log.md
│   ├── overview.md
│   ├── entities/
│   ├── concepts/
│   ├── sources/
│   ├── queries/
│   ├── comparisons/
│   └── synthesis/
├── .obsidian/              # 可选，Obsidian 兼容
└── .llm-wiki/              # 应用私有状态（自动生成）
    ├── project.json
    ├── ingest-queue.json
    └── lancedb/            # 启用 embedding 时
```

### 1.2 常见「非项目格式」

例如只有书籍章节的扁平/嵌套 Markdown：

```text
books/
├── 郑玉巧婴儿卷/07_6-7个月.../01-本月婴儿特点.md
└── 崔玉涛宝贝健康公开课/01-母乳....md
```

缺少 `purpose.md`、`schema.md`、`raw/sources/`、`wiki/` 时，**不能**直接挂到 `LLM_WIKI_PROJECT` 使用，需要先按下文初始化。

### 1.3 数据放哪里

Wiki 项目是**运行时数据**，不进 `llm_wiki-server` Git。推荐统一放在：

```text
~/overseas-github/llm_wiki_projects/<ProjectName>/
```

## 2. 创建新项目（逐步）

以下以育儿书为例，项目名 `ParentingBooks`。请把路径换成你的实际目录。

### 步骤 1：建目录骨架

```bash
PROJECT=~/overseas-github/llm_wiki_projects/ParentingBooks

mkdir -p "$PROJECT"/{raw/sources,raw/assets,wiki/{entities,concepts,sources,queries,comparisons,synthesis}}
```

也可用桌面版 **Create Project** 对话框自动创建（`cd upstream && npm run tauri dev`）。

### 步骤 2：写 `purpose.md`

说明项目定位、用户会问什么、材料来源。Ingest 时 LLM 会读这个文件决定提取方向。

参考：`llm_wiki_projects/CivilCareer/purpose.md`

育儿书示例要点：

- 目标：把育儿书章节变成可检索、可问答的知识库
- 用户画像：新手父母
- 典型问题：几个月加辅食、吐奶怎么办、睡眠规律等
- 材料来源：郑玉巧、崔玉涛等书籍章节 Markdown

### 步骤 3：写 `schema.md`

定义 wiki 页面类型和提取规则。Ingest 质量很大程度取决于此文件。

参考：`llm_wiki_projects/CivilCareer/schema.md`

育儿书可定义：

| Type | Directory | Purpose |
|------|-----------|---------|
| source | wiki/sources/ | 每章/每节摘要 + 标签 |
| concept | wiki/concepts/ | 喂养、睡眠、发育等可复用知识 |
| entity | wiki/entities/ | 疾病名、疫苗、营养素等 |
| scenario | wiki/scenarios/ | 「宝宝 6 个月不爱吃辅食」等场景页 |
| lesson | wiki/lessons/ | 警示：常见误区、禁忌 |

并写明 frontmatter 字段、问答检索优先级（scenario > lesson > concept > source）。

### 步骤 4：把原始材料放进 `raw/sources/`

```bash
BOOKS=~/overseas-github/llm_wiki_projects/books

# 先试几篇，验证 ingest 质量
cp "$BOOKS/郑玉巧婴儿卷/07_6-7个月的婴儿(180-209天)/01-本月婴儿特点.md" \
   "$PROJECT/raw/sources/"

# 批量复制（排除 00_ 开头的元数据文件）
find "$BOOKS" -name '*.md' ! -name '00_*' \
  -exec cp {} "$PROJECT/raw/sources/" \;
```

**建议：**

- 文件名用 **kebab-case 英文或拼音**，减少路径编码问题（如 `01-yue-ying-er-te-dian.md`）。
- 同一文件名不要互相覆盖；批量复制前检查 `raw/sources/` 是否重名。
- `00_处理进度.md`、`00_目录与抽取指引.md` 等元数据**不要**放入 `raw/sources/`。
- 原始书目录可保留不动；项目只引用复制后的副本。

### 步骤 5：自检清单

- [ ] `purpose.md`、`schema.md` 已写好
- [ ] `raw/sources/` 里至少有 1 个源文件
- [ ] `wiki/` 目录存在（可为空，ingest 后会填充）
- [ ] 未把 `books/` 原目录直接当作 `LLM_WIKI_PROJECT`

## 3. 配置 LLM（Ingest 必需）

Ingest 会调用大模型，必须配置 API。

```bash
cd ~/overseas-github/llm_wiki-server

cp overlay/config/llm.example.json overlay/config/llm.json
# 编辑 llm.json：apiKey、model、customEndpoint
```

`llm.json` 支持环境变量占位，例如：

```json
{
  "llmConfig": {
    "provider": "custom",
    "apiKey": "${LLM_API_KEY}",
    "model": "your-model",
    "customEndpoint": "https://api.example.com/v1/chat/completions"
  },
  "embeddingConfig": {
    "enabled": true,
    "apiKey": "${DASHSCOPE_API_KEY}",
    "endpoint": "https://dashscope.aliyuncs.com/compatible-mode/v1/embeddings",
    "model": "text-embedding-v4"
  }
}
```

导出环境变量：

```bash
export LLM_API_KEY="你的密钥"
export DASHSCOPE_API_KEY="你的 embedding 密钥"   # 可选，用于向量搜索
export LLM_WIKI_CONFIG=~/overseas-github/llm_wiki-server/overlay/config/llm.json
export LLM_WIKI_PROJECT=~/overseas-github/llm_wiki_projects/ParentingBooks
```

`llm.json` 含密钥，已被 `.gitignore` 规则覆盖（勿提交）。

## 4. 运行 Ingest

### 4.1 确保 CLI 已构建

```bash
cd ~/overseas-github/llm_wiki-server
./scripts/build-cli.sh    # 首次或代码更新后
```

### 4.2 先试 1 个文件

```bash
./scripts/llm-wiki ingest \
  "$LLM_WIKI_PROJECT/raw/sources/01-本月婴儿特点.md" \
  --project "$LLM_WIKI_PROJECT" \
  --config "$LLM_WIKI_CONFIG"
```

成功后会看到类似输出：

```text
[ingest] done — N wiki file(s) written
  wiki/sources/...
  wiki/concepts/...
```

检查 `wiki/index.md` 和 `wiki/sources/` 内容是否符合 `schema.md` 预期。

### 4.3 批量 Ingest

CLI 当前**一次处理一个源文件**。仓库提供可续跑的批量脚本：

```bash
export LLM_WIKI_PROJECT=~/overseas-github/llm_wiki_projects/ParentingBooks
# LLM 配置：复制样例后填入密钥（*.local.json 已 gitignore，勿提交）
export LLM_WIKI_CONFIG=~/overseas-github/llm_wiki-server/overlay/config/server.example.json
# 或：cp overlay/config/server.example.json overlay/config/server.minimax.local.json 后编辑
# 或自建 llm.json（见 §3，仅 ingest 用）

# 前台运行（可看实时输出）
~/overseas-github/llm_wiki-server/scripts/ingest-batch.sh

# 后台运行（推荐，181 篇约 6–10 小时）
nohup ~/overseas-github/llm_wiki-server/scripts/ingest-batch.sh \
  >> /tmp/llm-wiki-ingest-ParentingBooks.nohup 2>&1 &
```

脚本行为（`scripts/ingest-batch.sh`）：

- 遍历 `raw/sources/*.md`，逐个调用 `llm-wiki ingest`
- **已入库跳过**：若 `wiki/sources/<同名文件>` 已存在则 SKIP
- **失败即停**：任一文件 ingest 失败时脚本 `exit 1`；已成功入库与 SKIP 的文件不会重复处理。修复问题后**重新运行**即可从未入库文件继续
- 日志默认写入 `/tmp/llm-wiki-ingest-<项目名>.log`（可用 `INGEST_LOG` 覆盖）

**查看进度：**

```bash
# 当前处理到第几个 / 是否完成
grep -E 'SKIP|\[ingest\] done|FAILED|\[[0-9]+/' /tmp/llm-wiki-ingest-ParentingBooks.log | tail -15

# 已入库源文件数（目标 = raw/sources 下的 .md 总数）
ls ~/overseas-github/llm_wiki_projects/ParentingBooks/wiki/sources/ | wc -l
find ~/overseas-github/llm_wiki_projects/ParentingBooks/raw/sources -name '*.md' | wc -l

# 进程是否在跑
ps aux | grep ingest-batch | grep -v grep
```

完成后日志末尾出现：`==> done: ok=181 fail=0 total=181`。

**注意：**

- 章节多 = API 调用多，建议先 ingest **1～2 篇**验证质量再全量跑。
- 每个文件约 **2～4 分钟**（取决于 LLM 响应速度）。
- 若进程**卡住**（无 FAILED、长时间无新日志），可 `pkill -f ingest-batch` 后重新执行；已 SKIP 的文件不会重做。
- 若日志出现 **FAILED**，须先排查该篇 ingest 错误，再重跑；脚本不会跳过失败文件继续下一篇。
- PDF/Office 在 headless CLI 下需桌面 pdfium 或 `preprocess --copy-fallback`；纯 Markdown/txt 可直接 ingest。

### 4.4 可选：向量索引（混合搜索）

配置好 `embeddingConfig` 后：

```bash
./scripts/llm-wiki reindex --vectors \
  --project "$LLM_WIKI_PROJECT" \
  --config "$LLM_WIKI_CONFIG"
```

向量数据写入 `<project>/.llm-wiki/lancedb/`。

### 4.5 其他 CLI 命令

```bash
# 关键词搜索（不依赖 LLM）
./scripts/llm-wiki search "几个月添加辅食" --project "$LLM_WIKI_PROJECT" --top-k 10

# 扫描 raw/sources 清单（md5、大小）
./scripts/llm-wiki rescan --project "$LLM_WIKI_PROJECT" --json

# 纯文本预处理（txt/md 通常不需要）
./scripts/llm-wiki preprocess note.md -o /tmp/out.txt
```

## 5. 用 HTTP 服务浏览新项目

### 5.1 确保 UI 已构建

```bash
cd ~/overseas-github/llm_wiki-server
./scripts/apply-patches.sh
VITE_BACKEND=http VITE_API_TOKEN=e2e-test-token ./scripts/build-web.sh
```

`VITE_API_TOKEN` 需与下面 `LLM_WIKI_API_TOKEN` 一致，否则页面 API 请求鉴权失败。

### 5.2 启动服务（指向新项目）

```bash
# 若已有实例在跑，先停掉
pkill -f llm-wiki-server || true

export LLM_WIKI_PROJECT=~/overseas-github/llm_wiki_projects/ParentingBooks
export LLM_WIKI_API_TOKEN=e2e-test-token
export LLM_WIKI_BIND=127.0.0.1:8080
export LLM_WIKI_STATIC=~/overseas-github/llm_wiki-server/upstream/dist
export LLM_WIKI_CONFIG=~/overseas-github/llm_wiki-server/overlay/config/server.example.json
# 含 API Key 的本地副本（gitignore）：server.minimax.local.json
# 或 server.example.json + 环境变量 LLM_API_KEY

cd ~/overseas-github/llm_wiki-server
./overlay/server/target/release/llm-wiki-server
```

浏览器打开：**http://127.0.0.1:8080/**（修改 UI 后请 **Ctrl+Shift+R** 强刷）

### 5.3 HTTP 页面上能做什么

- **Search**：关键词搜索 ingest 后的 `wiki/`
- **Wiki 浏览**：阅读结构化页面
- **Graph**：wikilink 知识图谱
- **Chat**：基于 wiki 的 RAG 对话（需 `LLM_WIKI_CONFIG` + `LLM_API_KEY`；聊天记录存浏览器 localStorage）

**Chat 在哪里？** 打开项目后，**中间主区域**即为 Chat（不是左侧竖条上的单独图标）。左侧第一个文档图标 = Wiki 视图，默认显示 Chat。对话列表为空时，点击 Chat 左栏 **「新建对话」**。

若看到 **「打开项目」欢迎页** 而非文件树 + Chat，说明 HTTP 项目未加载成功——多半是 `VITE_API_TOKEN` 与 `LLM_WIKI_API_TOKEN` 不一致或未重新构建 UI。见 [开发与测试.md §Q7](./开发与测试.md#q7http-页面打开后看不到-chat-面板)。

健康检查：

```bash
curl "http://127.0.0.1:8080/api/v1/health?token=e2e-test-token"
```

切换项目 = 改 `LLM_WIKI_PROJECT` 并重启服务。

## 6. 对话提问

### 6.1 HTTP 页面 Chat（推荐日常使用）

确保 server 启动时带有 LLM 配置：

```bash
export LLM_API_KEY="你的密钥"          # 或通义 DASHSCOPE 等
export LLM_WIKI_CONFIG=~/overseas-github/llm_wiki-server/overlay/config/server.example.json
# 编辑 server.example.json 中的 model、customEndpoint，或复制为 llm.json

# 重启 llm-wiki-server（见 §5.2）
```

浏览器打开 http://127.0.0.1:8080/ → 中间 **Chat** 区域提问（先点「新建对话」若列表为空）。

**轻量版（推荐给家长等非技术用户）：** http://127.0.0.1:8080/lite/

- 选「育儿百科」或「职场进阶」即可对话（无需接触文件树与设置）
- 助手回复支持 **Markdown 渲染**（列表、粗体、代码块等）
- 流式过程中显示「正在检索资料… / 正在生成回答…」，结束后自动消失

详见 [overlay/static/lite/README.md](../overlay/static/lite/README.md)。  
流程：前端 RAG 检索 `wiki/` → 服务端 `POST /api/v1/projects/{id}/chat` 流式调用 LLM → 带引用回答。

构建 UI 时务必带上与 server 一致的 token：

```bash
VITE_BACKEND=http VITE_API_TOKEN=e2e-test-token ./scripts/build-web.sh
```

验证 Chat 是否就绪：

```bash
curl "http://127.0.0.1:8080/api/v1/runtime-config?token=e2e-test-token"
# chatEnabled: true 表示可对话
```

## 8. 常见问题

### Q1：`127.0.0.1 拒绝连接`

服务没在跑，不是端口被占。先 `ss -tlnp | grep 8080` 确认无监听，再按 §5.2 启动 `llm-wiki-server`。

### Q2：页面能开，搜索没结果

- 确认 `LLM_WIKI_PROJECT` 指向已 ingest 的项目
- 确认 ingest 已写入 `wiki/`（不只是 `raw/sources/` 有文件）
- 用 CLI 验证：`./scripts/llm-wiki search "关键词" --project "$LLM_WIKI_PROJECT"`

### Q3：Ingest 报错 / 无输出

- 检查 `LLM_API_KEY`、`model`、`customEndpoint`
- 先试单文件；看终端 `[ingest]` 日志
- Node 未安装会导致 ingest 失败（`build-cli.sh` 会装 node 依赖）

### Q4：材料已是 Markdown，还要 Ingest 吗？

要。`raw/sources/` 里的 Markdown 是**原材料**；检索和问答针对的是 LLM 编译后的 `wiki/` 结构（摘要、概念、场景页、交叉引用）。若只想全文检索原始 md，LLM Wiki 不是最合适工具。

### Q5：多个项目如何切换？

改 `LLM_WIKI_PROJECT` 环境变量并重启 HTTP 服务。每个项目独立目录，互不影响。

### Q6：页面能开，但看不到 Chat / 只有欢迎页？

- 确认已用 `VITE_API_TOKEN` 构建 UI，且与 `LLM_WIKI_API_TOKEN` 一致（见 §5.1）
- 确认 server 在跑：`curl "http://127.0.0.1:8080/api/v1/projects?token=e2e-test-token"`
- 强刷浏览器缓存（Ctrl+Shift+R）
- 完整排错见 [开发与测试.md §Q7](./开发与测试.md#q7http-页面打开后看不到-chat-面板)
