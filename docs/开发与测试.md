## 3. 环境与一次性准备

```bash
cd ~/overseas-github/llm_wiki-server
git submodule update --init --recursive

# Node（建议 nvm）
source ~/.nvm/nvm.sh

# Rust（cargo 已安装则跳过）
# protoc：build-cli.sh 会自动下载到 .tools/protoc（LanceDB 依赖）
```

### 环境变量速查

| 变量 | 用途 |
|------|------|
| `LLM_WIKI_PROJECT` | Wiki 项目根目录（含 `wiki/`、`raw/`） |
| `LLM_WIKI_API_TOKEN` | HTTP API 鉴权 token |
| `LLM_WIKI_BIND` | 监听地址，如 `127.0.0.1:8080` |
| `LLM_WIKI_STATIC` | 静态 UI 目录，即 **`upstream/dist`** 构建产物（见 [代码结构总览 §12.1](./代码结构总览.md#121-upstreamdist-构建产物)） |
| `LLM_WIKI_CONFIG` | 服务端 JSON（`overlay/config/server.example.json`） |
| `LLM_WIKI_REPO` | 集成仓库根路径；`./scripts/llm-wiki` 与 `ingest-batch.sh` 会自动设为仓库根（Chat 子进程找 `overlay/cli/node` 用） |
| `VITE_BACKEND=http` | 构建 HTTP 只读 UI 时由 `build-web.sh` 设置 |
| `VITE_API_TOKEN` | 构建 UI 时嵌入 bundle，需与 `LLM_WIKI_API_TOKEN` 一致 |

## 4. 构建

```bash
# 应用 patch（构建 HTTP UI 前必须）
./scripts/apply-patches.sh

# 一键：HTTP UI + server + CLI
./scripts/build-all.sh

# 分项
VITE_BACKEND=http VITE_API_TOKEN=your-token ./scripts/build-web.sh
cargo build --release --manifest-path overlay/server/Cargo.toml
./scripts/build-cli.sh
```

**注意：** 修改 `llm-wiki-common` 后需**同时**重建 server 与 CLI，否则 CLI 可能仍含旧逻辑（例如 search panic）。

## 5. 运行

### 4.1 Headless Server + 浏览器（HTTP 只读）

```bash
export LLM_WIKI_PROJECT=~/overseas-github/llm_wiki_projects/CivilCareer
export LLM_WIKI_API_TOKEN=e2e-test-token
export LLM_WIKI_BIND=127.0.0.1:8080
export LLM_WIKI_STATIC=~/overseas-github/llm_wiki-server/upstream/dist
export LLM_WIKI_CONFIG=~/overseas-github/llm_wiki-server/overlay/config/server.example.json

./overlay/server/target/release/llm-wiki-server
# 浏览器打开 http://127.0.0.1:8080/
```

健康检查：

```bash
curl "http://127.0.0.1:8080/api/v1/health?token=e2e-test-token"
```

### 4.2 CLI

```bash
export LLM_WIKI_PROJECT=~/overseas-github/llm_wiki_projects/CivilCareer

./scripts/llm-wiki search "职场" --project "$LLM_WIKI_PROJECT" --top-k 10
./scripts/llm-wiki rescan --project "$LLM_WIKI_PROJECT" --json
./scripts/llm-wiki preprocess note.md -o /tmp/out.txt

# ingest / 向量重建需 LLM 配置
cp overlay/config/llm.example.json overlay/config/llm.json
# 编辑 apiKey、model、endpoint
export LLM_WIKI_CONFIG=overlay/config/llm.json
./scripts/llm-wiki reindex --vectors --project "$LLM_WIKI_PROJECT"
./scripts/llm-wiki ingest source.pdf --project "$LLM_WIKI_PROJECT"
```

CLI 参数使用 **kebab-case**，例如 `--top-k`（不是 `--top_k`）。

### 4.3 桌面版（完整 LLM Chat / 入库 / 写入）

与官方 `llm_wiki` 体验一致：

```bash
cd upstream
npm install
npm run tauri dev
```

在 Settings 配置 LLM → Chat 面板对话 → RAG 检索 wiki。

### 4.4 Docker

把 **Web UI（upstream/dist）+ llm-wiki-server** 打成一体镜像，挂载宿主机 Wiki 目录。不是前端 dev 服务器，也不含 CLI ingest。

```bash
export LLM_WIKI_PROJECT=~/overseas-github/llm_wiki_projects/CivilCareer
export LLM_WIKI_API_TOKEN=e2e-test-token
docker compose -f docker/docker-compose.yml up --build
```

| 能力 | 容器内 |
|------|--------|
| 浏览 / 搜索 / 图谱 | ✅ |
| Chat | ⚠️ 当前 runtime 无 Node，可能不可用 |
| Ingest | ❌ 在宿主机跑 `llm-wiki ingest` |

WSL 下用 `./scripts/e2e-docker.sh`；详见 [docker/README.md](../docker/README.md)、[代码结构总览 §12](../docs/代码结构总览.md#122-docker-容器化部署)。

## 7. 常见问题与答复（FAQ）

### Q1：如何继续 Phase 2 / Phase 3 / Phase 4？

**答：** 均已在本仓库完成（见 §1）。Phase 4 将 upstream 升至 **v0.4.20** 并重生 `overlay/patches/0002-http-ui-bootstrap.patch`。后续升级：

```bash
./scripts/sync-upstream.sh vX.Y.Z
./scripts/apply-patches.sh
# patch 冲突时手动合并 upstream 文件后：
# cd upstream && git diff > ../overlay/patches/0002-http-ui-bootstrap.patch

```

### Q2：如何提交并推送改动？

**答：** 按 Git 安全流程：只提交 `overlay/`、`scripts/`、`docs/`，**不要**提交 `upstream/` 内 patch 脏工作区；submodule 仅更新指针到干净 tag。

```bash
cd upstream && git reset --hard v0.4.20 && cd ..
git add overlay/ scripts/ docs/ upstream
git commit -m "your message"
git push origin main
```

`upstream/` 的 HTTP patch 在**构建时**由 `./scripts/apply-patches.sh` 应用，不写入 submodule commit。

### Q3：如何做端到端验证（Docker + HTTP UI + 真实 wiki）？

**答：**

1. **本地（推荐）：** `./scripts/e2e-local.sh` 或 `./scripts/e2e-full.sh`
2. **Docker：** `./scripts/e2e-docker.sh`（需 Docker Desktop 运行且能拉取 `node`/`rust`/`debian` 镜像）
3. **手动：** 启动 server（§4.1）+ 浏览器访问 CivilCareer 项目

HTTP 模式下可验证：**浏览 wiki、关键词搜索、知识图谱、Chat RAG**（需 `LLM_WIKI_CONFIG`）。入库 / 写 wiki 不在此模式。

### Q5：像桌面 llm_wiki 那样，在**页面上**测试大模型 Chat 怎么做？

**答：** **HTTP 部署现已支持 Chat + RAG**（2026-06-06 起），通过服务端 LLM 代理绕开浏览器 CORS。

| 能力 | HTTP UI | 桌面 `tauri dev` |
|------|---------|------------------|
| 浏览 / 搜索 / 图谱 | ✅ | ✅ |
| Chat 流式对话（RAG） | ✅（`POST /chat` SSE + `LLM_WIKI_CONFIG`） | ✅ |
| Settings 测试 LLM 连接 | ⚠️ 使用服务端配置，非浏览器直连 | ✅ |
| 入库 / 写 wiki | ❌ | ✅ |
| Chat 附加 Web Search | ❌（首期） | ✅ |

**HTTP Chat 启动：**

```bash
# 1. 构建 UI（token 在构建时写入 bundle）
VITE_BACKEND=http VITE_API_TOKEN=e2e-test-token ./scripts/build-web.sh

# 2. 启动 server
export LLM_API_KEY="你的密钥"
export LLM_WIKI_API_TOKEN=e2e-test-token          # 与 VITE_API_TOKEN 一致
export LLM_WIKI_CONFIG=overlay/config/server.example.json   # 含 llmConfig
export LLM_WIKI_STATIC=upstream/dist
# 编辑 model / customEndpoint 后重启 server

curl "http://127.0.0.1:8080/api/v1/runtime-config?token=$LLM_WIKI_API_TOKEN"
# chatEnabled: true → 打开 http://127.0.0.1:8080/
```

**界面说明：** Chat 不是单独导航项，而是打开项目后**中间主区域**（左侧第一个 Wiki 图标视图）。对话列表为空时点击 **「新建对话」**。修改 UI 或 token 后需重新构建并 **Ctrl+Shift+R** 强刷。

聊天记录在 HTTP 模式下存 **localStorage**（非 `.llm-wiki/chats/`）。

**Lite 轻量页（推荐给非专业人士）：** http://127.0.0.1:8080/lite/

- 纯静态 HTML + ES module，无 React 运行时
- 首页选知识库主题（`projects[]` 多项目）；左侧历史侧栏；RAG + SSE 流式对话
- **流式状态：**「正在检索资料…」→「正在生成回答…」，结束后标示消失；120 秒超时
- **Markdown：** 助手回复经 `marked` + `DOMPurify` 渲染（标题、列表、代码、表格）
- 构建：`build-web.sh` 复制到 `dist/lite/` 并同步 `vendor/` 依赖

详见 [overlay/static/lite/README.md](../overlay/static/lite/README.md)。Lite 排错见 **Q8**。

**桌面版（入库 + 写入）：** `cd upstream && npm run tauri dev`

**测 LLM 入库流水线：** `./scripts/llm-wiki ingest ...` 或 `./scripts/ingest-batch.sh`（见 [新项目指引.md §4.3](./新项目指引.md#43-批量-ingest)）

**育儿书示例项目 `ParentingBooks`：** 材料来自 `llm_wiki_projects/books/`（181 篇），批量入库脚本与进度查看见 [新项目指引.md §10](./新项目指引.md#10-示例从-books-到-parentingbooks)。

### Q6：调试技巧

| 现象 | 处理 |
|------|------|
| CLI search panic（中文） | 重建 CLI：`cargo build --release --manifest-path overlay/cli/rust/Cargo.toml` |
| `GET /files` 返回 413 | 大项目 listing 超过 `maxFiles`；用 `recursive=false` 或增大 `maxFiles` |

| **看不到 Chat，只有欢迎页** | 见 **Q7** |
| **Lite `/lite/` 首页无项目卡片** | 见 **Q8** |

### Q7：HTTP 页面打开后看不到 Chat 面板？

**答：** 常见原因是 **项目未自动加载**，页面停在「打开 / 创建项目」欢迎屏，或 API 鉴权失败。

**正常界面：** 左侧文件树 + **中间 Chat** + 可选右侧预览。Chat 无单独侧边栏图标。

**排查步骤：**

1. **确认 server 与项目**

   ```bash
   curl -H "Authorization: Bearer $LLM_WIKI_API_TOKEN" http://127.0.0.1:8080/api/v1/projects
   # 应返回 currentProject（来自 LLM_WIKI_PROJECT）
   ```

2. **确认 UI 构建带 token 且使用 overlay 模块**

   ```bash
   VITE_BACKEND=http VITE_API_TOKEN=e2e-test-token ./scripts/build-web.sh
   grep -r "e2e-test-token" upstream/dist/assets/ | head -1   # 应有输出
   ```

3. **确认 Chat 已启用**

   ```bash
   curl "http://127.0.0.1:8080/api/v1/runtime-config?token=$LLM_WIKI_API_TOKEN"
   # chatEnabled: true
   ```

4. **浏览器强刷** `Ctrl+Shift+R`（避免旧 `dist` 缓存）。

5. **进入对话：** 中间 Chat 左栏点 **「新建对话」** 再提问。

**技术原因（2026-06-06 修复）：** `vite.config.ts` 中若通用别名 `@` 排在 overlay 路径（如 `@/commands/fs`）之前，HTTP 构建会误用上游 Tauri 版 `fs.ts`，其中 `bootstrapHttpProject()` 恒返回 `null`，导致永远不加载 server 项目。overlay 别名必须写在 `@` **之前**。详见 [overlay/web/README.md](../overlay/web/README.md)。

日志：

```bash
# e2e-local 后台 server 日志
/tmp/llm-wiki-e2e.log

# 手动启动时可前台看 stderr
RUST_BACKTRACE=1 ./overlay/server/target/release/llm-wiki-server
```

### Q8：Lite 轻量页 `/lite/` 首页没有项目卡片，或一直加载？

**答：** Lite 使用 ES module（`app.js` → `markdown.js` → `vendor/*.js`）。任一脚本加载失败会导致 `init()` 不执行，首页 `#project-grid` 为空。

**常见原因与处理：**

1. **静态资源未同步** — 修改源码后需复制到 `dist`：

   ```bash
   cp -r overlay/static/lite/. upstream/dist/lite/
   # 浏览器 Ctrl+Shift+R
   ```

2. **缺少 `vendor/`** — Markdown 依赖未复制。应存在：

   ```
   upstream/dist/lite/vendor/marked.esm.js
   upstream/dist/lite/vendor/purify.es.js
   ```

   推荐用完整构建（会自动从 `node_modules` 复制 vendor）：

   ```bash
   VITE_BACKEND=http VITE_API_TOKEN=e2e-test-token ./scripts/build-web.sh
   ```

3. **`.mjs` MIME 类型（已修复）** — Lite 仅使用 `vendor/purify.es.js`（勿保留 `.mjs` 副本）；server `static_files.rs` 也为 `.mjs` 设置了 `application/javascript`。

4. **API 鉴权失败** — 与完整版相同，`config.js` 的 `apiToken` 须与 `LLM_WIKI_API_TOKEN` 一致：

   ```bash
   curl "http://127.0.0.1:8080/api/v1/projects?token=$LLM_WIKI_API_TOKEN"
   ```

5. **多项目未配置** — 需 `projects[]` 或 `LLM_WIKI_PROJECTS`，否则 banner 提示「暂无可用知识库」。

**流式 / Markdown 相关：**

| 现象 | 说明 |
|------|------|
| 长时间「正在检索资料…」 | 正常，RAG search 完成前无 LLM token；之后变为「正在生成回答…」 |
| 回复结束仍显示「正在回复…」 | 不应出现；若出现请查 Console 是否在 `finally` 前抛错 |
| 回复显示 `**markdown**` 原文 | `markdown.js` 或 `vendor/` 未加载，按上表排查 |
| 120 秒超时 | 单次问答保护；可检查 LLM 配置与 Node `cmd-llm-stream.ts` |

完整 Lite 说明：[overlay/static/lite/README.md](../overlay/static/lite/README.md)
